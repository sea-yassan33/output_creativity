# 増幅器かイコライザーか？ソフトウェアエンジニアリングのプロジェクトベース学習におけるLLMの進化に関する長期研究

## 1.導入

プロジェクトベース学習（PBL）は、ソフトウェアエンジニアリング（SE）教育において広く採用されているアプローチです。SE手法に従ってソフトウェア成果物の作成に学生を参加させることで、PBLは学生が理論的な知識を応用し、要件定義からテストまでの開発ライフサイクル全体を経験するための実践的なコンテキストを提供します。学生が自ら問題と解決策を定義する探究型PBL（IPBL）は、あらかじめ仕様が決められたプロジェクトと比較して、より大きな内発的動機と主体性を育むことができるため、特に高く評価されています 。

同時に、大規模言語モデル（LLM）の驚異的な機能によってソフトウェア開発の状況が再構築されつつあります。コード生成やその他のSE活動における熟練度を考慮すると、LLMはソフトウェア業界全体で急速に採用されており、LLMツールを活用する能力はソフトウェアエンジニアにとって不可欠な能力になりつつあるこの業界の変化は、SE教育にLLM強化活動を単なる付加的なものではなく、学生の将来のキャリア準備の不可欠な部分として進化させ、組み込むという明確な義務を課してこの業界の変化は、SE教育にLLM強化活動を単なる付加的なものではなく、学生の将来のキャリア準備の不可欠な部分として進化させ、組み込むという明確な義務を課している。

しかし、入門プログラミングにおけるLLMの教育的利用はまたは特定のSEタスクは研究されてきましたが、SE PBL の文脈におけるそれらの影響は、依然としてほとんど不明です。明確に定義されたプログラミング演習や孤立したタスクとは異なり、PBL は学生に大きな自由を与え、LLM のより高度で適応的な使用を要求します。この固有の複雑さにより、特に LLM の進化が継続的に進んでいることを考慮すると、PBL に LLM を統合する際の教育設計の難しさが増します。LLM の急速な進化は、PBL における学生のパフォーマンスをどのように変えてきたのでしょうか。最新の LLM はどのような新しい機会を解き放ち、どのような新しい課題をもたらすのでしょうか。

この目的のために、本稿では、学部3年生のソフトウェアエンジニアリング（SE）コースにおける5週間のIPBL課題を基盤とした、2年間の縦断的実証研究を紹介します。具体的には、学生が主に無料で公開されているLLMを使用した2024年度のコホートと、より高性能な有料LLMを使用した2025年度のコホートを比較します。これら2つのグループのパフォーマンスを分析することで、SE PBLにおけるLLMの急速な進化がもたらす多面的な影響に関する最初の実証的証拠を提供することを目指します。

## 2.研究デザインと背景

コースの背景。この研究は、地元の大学のコンピュータサイエンス学部3年生を対象とした、必修の15週間のSEコースに基づいて実施されました。参加者全員が、プログラミング、データ構造、アルゴリズムの基礎コースを修了し、開発に必要な技術スキルを習得しました。このコースは、ソフトウェア開発ライフサイクルの基本に関する10週間の講義と、それに続く5週間の個別IPBLで構成され、学生は各自が選択した完全なアプリケーションを開発します。学生はウォーターフォールモデルそしてすべての中間成果物を保持します。コースの内容と講師は 2024 年と 2025 年の間で変更されませんでしたが、学生の LLM の使用状況は変化しました。2024 年の学生は主に無料の ChatGPT 4o を使用しましたが、2025 年の学生は主に高機能の Gemini 2.5 Pro (学生向けの Pro プラン) を使用しました。

データ収集と測定。我々は、(i) COSMIC 機能ポイント (CFP) を評価します。これはソフトウェアの機能サイズを捉え、SE で 20 年以上の教育経験を持つコースインストラクターによって公式の COSMIC プロトコルを使用して測定されます。IPBL の目的はアプリケーション プロトタイプを開発することであるため、この指標を学生のパフォーマンスを測定する主要な指標として使用します。(ii) CFRP (プログラミング スキルの共通参照フレームワーク) の事前および事後。これは、学生のコア プログラミング能力を 6 レベルで測定します。テストケース合格率：各プロジェクトにつき24（通常8、境界8、例外8）の単体テストケースを使用して測定され、これらのテストケースは構造化されたプロンプトを使用してLLMで生成されます。

## 4.議論と示唆

機会: LLM は平等化装置として機能する。機能規模の劇的な増加とプログラミング能力からの分離が示唆するように、優れた LLM はある程度平等化装置として機能し、ほとんどの学生、特にプログラミングが苦手な学生に、真の SE の課題が現れるプロジェクトの複雑さのレベルに到達するための手段を提供することで機会を提供すると考えています。従来、 5 週間の IPBL では、学生は労力の大部分を基本的な機能のコーディングに費やし、高度な SE プラクティスのより深い適用にはほとんど時間が残されていませんでした。このような IPBL の主な目標は、学生に SE の概念と開発フェーズに慣れてもらうことですが、手動実装の実際的な制約により、高度な SE プラクティス (アーキテクチャ設計など) が抽象的または「おもちゃ」プロジェクトには不要に感じられるほど範囲が制限されることがよくありました。高度な LLM は、この方程式を根本的に変えます。ルーチン的なコード生成の多くを自動化することで、学習ベースのプロジェクトが、限られた時間枠内で、単なる概念的な理解を超えて、より本格的で複雑なエンジニアリングの実践に進むことを可能にします。たとえば、モジュール設計は、より大きなコードベースで具体的に実現可能になります。迅速なプロトタイピングは、真のアジャイル開発手法を可能にします。この点において、優れたLLM（学習学習モデル）は、PBL（問題解決型学習）と同じ時間的制約の中で、より現実的で高度なソフトウェアエンジニアリングの実践を提供する機会をもたらします。

課題：LLMは増幅器として機能する。プロジェクトの複雑さを民主化する一方で、より高度なLLMへの移行は、絶対的なパフォーマンス格差の爆発的な拡大に見られるように、不平等のより強力な増幅器として同時に機能した。これは、介入がなければ、LLMは高パフォーマンスの学生の優位性をさらに強化する一方で、遅れてスタートした学生はさらに疎外され、LLMを効果的に活用できないことからフラストレーションを感じる可能性があることを意味する。決定的な因果分析には今後の研究でより詳細な情報を収集する必要があるが、私たちの教育経験は、LLM強化SE時代に必要とされる重要なスキルの変化が原因であることを示唆している。これには、システム思考と問題分解能力、LLMを使用する際の迅速なエンジニアリングと反復的な改良能力、LLMと協力する際の批判的評価と統合能力が含まれる可能性がある。この意味で、優れた LLM の導入は 2 つの重要な課題をもたらします。(i) 戦略的な問題分解や反復的なプロンプト最適化など、 LLM を使用する能力が「隠れたカリキュラム」となり、新たな教育格差を生み出しています。(ii) SE 教育の焦点は、従来の SE 知識を単独で教えることから、効果的な人間と LLM のコラボレーションパターンなど、LLM で強化された SE 実践を教える方向に進化する必要があります。
