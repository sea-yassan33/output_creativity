## 0.使用するツール

### 【Socket.dev:パッケージインストール前の確認ツール】

- Socket.dev
  - https://socket.dev/

- 確認する観点
  - インストール時に勝手に実行されるスクリプト（postinstall）があるか。
  - ネットワーク通信を行うか、メンテナンスが止まっていないか。

### 【OSV-Scanner：プロジェクト毎の脆弱性診断ツール】

- windowsでインストールする場合

```sh
## パッケージの検索（確認）
winget search Google.OSVScanner

## インストール

## バージョンの確認(ターミナル再起動後)

```

## 1.脆弱性診断フロー

- 脆弱性診断フローの例

```
[ 新パッケージ導入検討 ]
        ↓
  ① Socket.dev で事前調査
    （npm install 実施前）
        ↓
  ② npm install / 開発・実装
        ↓
  ③ OSV-Scanner でプロジェクト全体スキャン
    （リリース前・PR時）
        ↓
  ④ Highリスク以上 → 調査・対策
        ↓
  [ リリース ]
```

- 各フェーズの仕様ツール

| フェーズ | ツール | 目的 |
|---|---|---|
| パッケージ導入前 | Socket.dev | 導入予定パッケージの安全性評価 |
| リリース前 | OSV-Scanner | プロジェクト全体の既知脆弱性スキャン |

## 2.パッケージインストール前に調査【Socket.dev】

Socket.dev は、npmパッケージのサプライチェーンリスクを可視化するサービスです。単なる既知CVEの検出にとどまらず、パッケージの**振る舞い・意図的な悪意**まで分析します。

- 公式サイト: [https://socket.dev](https://socket.dev)

### 2-2. 調査手順

#### Webサイトで調査

1. 下記の各サイトで調べる

```
## node
https://socket.dev/npm/package/＜パッケージ名＞
https://deps.dev/npm/＜パッケージ名＞

## python
https://socket.dev/pypi/package/＜パッケージ名＞
https://deps.dev/pypi/＜パッケージ名＞

## 例（node）
https://socket.dev/npm/package/flatted
https://socket.dev/npm/package/flatted/overview/3.3.3
```

2. スコアとリスク項目を確認する



### 2-3. 見るべき観点（Socket.dev）

Socket.devのスコアは複数カテゴリで構成されています。以下の観点を重点的に確認してください。

#### ① Supply Chain Risk（サプライチェーンリスク）⭐ 最重要

| チェック項目 | 内容 | 危険なサイン |
|---|---|---|
| **Install scripts** | `preinstall` / `postinstall` スクリプトの有無 | スクリプトが存在し、内容が不明瞭 |
| **Telemetry** | インストール時に外部送信する通信の有無 | 予期しない外部通信がある |
| **Obfuscated code** | コードが難読化されているか | 難読化あり |
| **Dynamic code execution** | `eval()` や `Function()` の動的実行 | 利用あり |
| **Network access** | パッケージが意図しないネットワーク通信を行うか | 不審なドメインへの接続 |

#### ② Vulnerability（既知脆弱性）

| チェック項目 | 内容 |
|---|---|
| **CVE件数** | 報告されているCVEの数とSeverity |
| **未パッチ期間** | 脆弱性が発見されてからパッチリリースまでの日数 |

#### ③ Quality（品質・信頼性）

| チェック項目 | 内容 | 懸念サイン |
|---|---|---|
| **Maintenance** | 最終更新日・メンテナンス状況 | 2年以上更新なし |
| **Deprecated** | 公式に非推奨とされていないか | 非推奨マークあり |
| **Unpopular package** | ダウンロード数が著しく少ない | 週次DL数が100未満 |
| **No repository** | GitHubリポジトリが存在するか | リポジトリ未設定 |

#### ④ License（ライセンスリスク）

| チェック項目 | 内容 | 懸念サイン |
|---|---|---|
| **License type** | MITやApache 2.0などOSSライセンス | GPL系（商用利用制限あり） |
| **License conflict** | 依存関係間でライセンス競合がないか | 競合あり |

### 2-4. 導入判断基準

```
Socket Score 80以上      → 導入OK
Socket Score 60〜79      → 懸念項目を確認の上、チームで判断
Socket Score 60未満      → 原則導入見送り・代替パッケージを検討
Supply Chain Riskが高い  → スコアに関わらず導入禁止（要レビュー）
```

## 3.脆弱性スキャン

### 3-1. ツール概要

OSV-Scanner は Google が開発するOSSの脆弱性スキャンツールです。[OSV（Open Source Vulnerabilities）データベース](https://osv.dev) を参照し、プロジェクトのロックファイルや依存関係から既知の脆弱性を検出します。

### 3-2.脆弱診断

#### 基本スキャン

```sh

# プロフェクトに移動
cd /path/to/your/project

# カレントディレクトリのロックファイルを自動検出してスキャン
osv-scanner scan --recursive .

# 特定のディレクトリを指定してスキャン
osv-scanner --path /path/to/your/project

# package-lock.json を直接指定
osv-scanner scan --lockfile package-lock.json

# yarn.lock を指定
osv-scanner scan --lockfile yarn.lock

# 複数ファイルを指定
osv-scanner scan \
  --lockfile package-lock.json \
  --lockfile path/to/other/package-lock.json
```

#### 出力形式について

```sh

# JSON形式で出力（CI/CDパイプラインでのパース用）
osv-scanner scan --recursive . --format json > osv-result.json

# テーブル形式（デフォルト・視認性が高い）
osv-scanner scan --recursive . --format table

# SARIF形式（GitHub Advanced Security連携用）
osv-scanner scan --recursive . --format sarif > osv-result.sarif
```

#### ① Severity（深刻度）⭐ 最重要

| レベル | CVSSスコア | 対応優先度 |
|---|---|---|
| **Critical** | 9.0〜10.0 | 🔴 即時対応必須（リリースブロック） |
| **High** | 7.0〜8.9 | 🟠 リリース前に対応必須 |
| **Medium** | 4.0〜6.9 | 🟡 次スプリントまでに対応 |
| **Low** | 0.1〜3.9 | 🟢 計画的に対応 |

#### ② 影響範囲（Direct vs Transitive）

| 種別 | 説明 | 優先度 |
|---|---|---|
| **Direct dependency** | `package.json` に直接記載されているパッケージ | 高 |
| **Transitive dependency** | 依存パッケージがさらに依存しているパッケージ | 中（対応難易度も考慮） |

#### ③ 脆弱性の種類（CWE）

**【以下のCWEカテゴリが含まれる場合は特に注意】**

| CWE | 内容 |
|---|---|
| CWE-79 | XSS（クロスサイトスクリプティング） |
| CWE-89 | SQLインジェクション |
| CWE-22 | パストラバーサル |
| CWE-502 | 信頼できないデータのデシリアライズ |
| CWE-400 | リソース枯渇（DoS） |
| CWE-918 | SSRF（サーバーサイドリクエストフォージェリ） |

- CWEは大きく以下のカテゴリに分類されており、番号帯でおおよその種別がわかります。

| 番号帯 | カテゴリ |
|---|---|
| CWE-20〜99 | 入力検証・エンコード関連 |
| CWE-200〜299 | 情報漏洩 |
| CWE-300〜399 | 認証・通信関連 |
| CWE-400〜499 | リソース管理 |
| CWE-600〜699 | 言語固有の問題（Java, PHPなど） |
| CWE-700〜799 | 設計・アーキテクチャ上の問題 |
| CWE-1200〜1399 | ハードウェア・最新Web技術（**1321はここ**） |

- CWEコードの意味を調べる方法

```
https://cwe.mitre.org/data/definitions/＜番号＞.html
```

#### ④ 修正バージョンの有無

```
修正バージョンあり  → アップデートで対応（最優先）
修正バージョンなし  → 回避策（workaround）または代替パッケージ検討
```

## 4. Highリスク以上検出時の対応手順

OSV-Scannerでリスクが **High（CVSSスコア7.0以上）またはCritical** の脆弱性が検出された場合、以下の手順で調査・対策を進めてください。

### 【対策フロー】

```
① パッケージのアップデート（最優先）
        ↓ 修正バージョンがない場合
② 代替パッケージへの移行
        ↓ すぐに移行できない場合
③ 回避策（workaround）の実装
        ↓ どれも難しい場合
④ リスク許容（根拠を記録して期限付きで管理）
```

> 【Pythonの場合】

```sh
# ① OSV-Scannerで脆弱パッケージを特定（例: urllib3）
pip freeze > requirements.txt
osv-scanner scan --lockfile requirements.txt --format json > osv-result.json

# ② 直接インポートしているか確認
grep -rn -e "import urllib3" -e "from urllib3" ./src

# ③ 間接依存を確認（誰経由で使われているか）
## pip install pipdeptree
pipdeptree --reverse --packages urllib3

# ④ 該当箇所で脆弱な機能を使っているか確認
#    例: urllib3のSSL検証無効化
grep -rn "verify=False" ./src

```

### **【STEP1】** json形式でスキャン

```sh
# 【npmの場合】
# JSON形式で出力（CI/CDパイプラインでのパース用）
osv-scanner scan --recursive .  --format json > osv-result.json

# 【pythonの場合】
pip freeze > requirements.txt
osv-scanner scan --lockfile requirements.txt --format json > osv-result.json

```

### **【STEP2】** 出力したファイルをDataFrame形式→CSV出力

- 下記のツールを参考に出力結果をまとめる
  - [main.py](./診断ツール/sample_main.py)
  - [my_func.py](./診断ツール/my_func.py)

### **【STEP3】** 一次情報源で詳細を確認する

| 調査先 | URL | 確認内容 |
|---|---|---|
| **NVD（米国国家脆弱性データベース）** | `https://nvd.nist.gov/vuln/detail/CVE-XXXX-XXXXX` | CVSSスコア・攻撃ベクター・影響範囲 |
| **OSV.dev** | `https://osv.dev/vulnerability/CVE-XXXX-XXXXX` | 影響バージョン範囲・修正バージョン |
| **GitHub Advisory DB** | `https://github.com/advisories/GHSA-xxxx-yyyy-zzzz` | 修正PR・PoC・workaround情報 |
| **パッケージのChangelog/Release Notes** | npmページ or GitHubリポジトリ | 修正内容の確認 |

**NVDの見方（重要指標）:**

```
Attack Vector (AV):     NETWORK = リモートから攻撃可能（最も危険）
                        LOCAL   = ローカル実行が必要
Attack Complexity (AC): LOW     = 特別な条件不要（最も危険）
                        HIGH    = 特定条件が必要
Privileges Required:    NONE    = 認証不要（最も危険）
User Interaction:       NONE    = ユーザー操作不要（最も危険）
```

### **【STEP4】** 自プロジェクトへの影響を評価する

- 該当コードの利用箇所を特定する

> node

```sh
# 脆弱なパッケージの利用箇所を検索
grep -rn "require('axios')" ./src
grep -rn "from 'axios'" ./src

# 依存ツリーで影響パッケージを確認
npm ls axios
# または
npm why axios
```

> python

```sh
# 使用しているパッケージを確認
grep -rn "nltk" ./project


```

- 脆弱性の「攻撃経路」が自プロジェクトで成立するか確認する
  - 以下の観点で**実際に悪用されるリスクがあるか**を判断

| 確認ポイント | 観点 |
|---|---|
| **露出面** | 脆弱な機能がエンドユーザーから呼び出せる状態か？ |
| **入力制御** | 脆弱性をトリガーする外部入力がアプリに存在するか？ |
| **実行環境** | クライアントサイドのみ / サーバーサイドのみ？ |
| **認証要件** | 攻撃に認証が必要か？自プロジェクトで認証済みユーザーのみか？ |

- 下記を例に調査・評価内容を記録する

### **【STEP6】** 対策を実施する

#### **対策A**: パッケージのアップデート（最優先）

修正バージョンが存在する場合はアップデートが最も確実な対策です。

> node

```bash
# 修正バージョンへアップデート
npm update <パッケージ名>

# 特定バージョンへ固定
npm install <パッケージ名>@<修正バージョン>

# アップデート後に動作確認
npm test
npm run build

# アップデート後に再スキャンして解消を確認
osv-scanner scan --recursive .
```

> python

```sh
# 特定バージョンへアップデート
pip install "パッケージ名==修正バージョン"

# 例: requests の脆弱バージョンを修正版へ
pip install "requests==2.32.0"

# requirements.txt を更新
pip freeze > requirements.txt

# アップデート後に再スキャンして解消確認
pip freeze > requirements.txt
osv-scanner scan --lockfile requirements.txt --format json > osv-result.json

# 削除が必要な場合
pip uninstall nltk


```

**メジャーバージョンアップの場合の注意:**

```sh
# Breaking Changeを確認
# 1. パッケージのCHANGELOGを確認
# 2. マイグレーションガイドを確認
# 3. テストが通ることを確認してからコミット

# 例: axios v0.x → v1.x の場合
# https://github.com/axios/axios/blob/v1.x.x/CHANGELOG.md を確認
```

#### **対策B**: npm overrides / resolutions（間接依存の場合）

脆弱なパッケージが**Transitive dependency（間接依存）**の場合、直接アップデートできないため `overrides` を使います。

> node

```json
// package.json（npm 8.3以降）
{
  "overrides": {
    "脆弱なパッケージ名": "修正済みバージョン"
  }
}
```

```json
// package.json（Yarn Berry / yarn.lock）
{
  "resolutions": {
    "脆弱なパッケージ名": "修正済みバージョン"
  }
}
```

```sh
# overrides設定後にインストール
npm install

# 解消確認
osv-scanner scan --recursive .
```

> python

```sh
# requirements.txt
## 脆弱なパッケージ名==修正バージョン   # 直接固定して上書き

pip install -r requirements.txt
```

> ⚠️ **注意:** overridesによる強制バージョン変更は、親パッケージの動作を壊す可能性があります。回帰テストを必ず実施してください。

#### **対策C**: 代替パッケージへの移行

修正バージョンが存在しない、またはメンテナンスが放棄されている場合、代替パッケージへの移行を検討します。

> node

```sh
# 調査手順
# 1. Socket.devで代替パッケージのスコアを確認
# 2. npm trendsで利用動向を確認（https://npmtrends.com）
# 3. GitHubスター数・最終コミット日・Issue対応状況を確認

# 例: 非推奨の request → axios または node-fetch へ移行
npm uninstall request
npm install axios
```

> python

```sh
# 旧パッケージを削除して代替をインストール
pip uninstall 旧パッケージ名
pip install 代替パッケージ名
```
